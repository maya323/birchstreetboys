Birch Street Boys Heuristic Evaluation

By Zoë McGinnis, James Loh, Julian Montes, and Claire Hashizume


Overall Evaluation

We felt that this design used color and size in a compelling way to emphasize important affordances such as in the “RSVP” and “PUBLISH” buttons. Another strong aspect of the design is the inclusion of necessary buttons and information to allow the user to interact with the design and accomplish various tasks, such as finding events and event details, creating an event, and favoriting events. Major weaknesses include the consistency of design elements across different pages, the lack of “close” or “back” buttons and ways for users to recognize and correct errors, and missing additional ways to navigate to different pages without needing to go through the hamburger menu.

Issues with the Heuristic

Below are the issues our team found with the prototype. The number of evaluators who identified each issue in their individual reports is listed after the severity rating.

General

  1. [H3 User Control and Freedom] (Severity 4) (Evaluators 2)

    There is no way to close the menu once the user has opened it. I would recommend including an X somewhere visible or enabling the user to close it by clicking the menu icon again.


  2. [H4 Consistency and Standards] (Severity 1) (Evaluators 1)

    The menu icon only has two bars, when I believe conventionally they usually have three. Just a small detail, but this tweak might make the interface look slightly more professional.


  3. [H4 Consistency and Standards] (Severity 1) (Evaluators 1)

    The hamburger menu covers half the screen whenever you open it. Sites that use hamburger menus that cover a substantial portion of the screen usually use a semi-transparent background to focus the user on the menu options being displayed (at least in my experience).


  4. [H7 Flexibility and Efficiency of Use] (Severity 3) (Evaluators 1)

    There is no immediate way to search events by keyword. Including a search bar at the top of the page or in the menu might enable users who are looking for information about a certain event to easily find that event.


  5. [H8 Aesthetic and minimalist design] (Severity 0) (Evaluators 1)

    Throughout the entire prototype, there are mostly neutral colors which don’t represent the limitless possibilities of this platform. There should be more colors that aren’t neutral. For example, look at the Google logo which has various colors on it, the history behind the colors was from lego bricks which can symbolize creativity.


  6. [H7 Flexibility and efficiency of use] (Severity 2) (Evaluators 1)

    At the header of the webpage is “Events this Month”, however, I think there should also be an option to select events in the present day if the user is bored and immediately thinks to themselves, “what am I going to do today”?


    Login

  7. [H2 Match between system and real world] (Severity 3) (Evaluators 1)

    On screen 1, users aren’t aware that the login uses SSO, so when they only see a login button and not an option to sign up, they may get confused and leave the page. The button should probably mention that users can login with SSO.


  8. [H2 Match between system and real world] (Severity 4) (Evaluators 1)

    On the login page (screen 2), the user is prompted for a username and password, however, SSO’s use an email instead. Users should not have to remember or make a new username when signing in with SSO.


    Create an Event

  9. [H3 User Control and Freedom] (Severity 3) (Evaluators 3)

    There does not appear to be any sort of cancel or back button that would allow the user to leave the page besides the menu. I would include a cancel button next to the publish button.


  10. [H9 Help users recognize, diagnose, and recover from errors] (Severity 2) (Evaluators 3)

    There is no indication as to which fields are mandatory and which are not, which may confuse the user in future iterations when the UI is more developed. We suggest putting a red asterisk next to required fields.


  11. [H1 Visibility of System Status] (Severity 3) (Evaluators 2)

    There is no header indicating what page the user is on. I would recommend including a header at the top of the page like “Create a New Event.”


  12. [H4 Consistency and Standards] (Severity 4) (Evaluators 1)

    It is unclear whether the “what is it,” “when/where,” and “Who’s going” fields are text or some other type of input. If they are text, I suggest reformatting them to match the style of the Enter Club Name and Enter Event Name boxes.


  13. [H5 Error prevention] (Severity 3) (Evaluators 1)

    On the create event page (screen 7), there is no confirmation after publishing. Having another step before publishing the event will help users identify areas of errors.


  14. [H4 Consistency and standards] (Severity 4) (Evaluators 1)

    Typically, when posting about an event or group, the organizer doesn’t need to include a list of attendees- people planning to attend individually RSVP, and some platforms will show that on the event detail page. Requiring the organizer to list who’s going on screen 7 when creating an event may require too much activation energy than they are used to.


  15. [H4 Consistency and standards] (Severity 2) (Evaluators 1)

    The create events page (screen 7) asks for a club name, but the event information page (screen 6) has the organizer name instead. There is inconsistency in what the page displays in the top right corner, whether users should put the club’s name or the organizer name.


    Event Description


  16. [H4 Consistency and standards] (Severity 4) (Evaluators 2)

    After RSVPing to an event, on screen 6, it goes directly to the favorited events page. Users may be confused as to whether they RSVPed since they didn’t get a confirmation for it. In addition, favorited events should be different from RSVPed events.


  17. [H2 Match between system and real world] (Severity 3) (Evaluators 1)

    On screen 6, there is a bell icon next to the organizer name. The bell icon can be misleading, as it’s usually used for notifications. A different icon such as a person could be used instead to indicate the organizer.

  18. [H3 User control and freedom] (Severity 3) (Evaluators 1)

    After clicking an event on the “Events this Month” page (screen 3), users currently don’t have an option to go back to the previous page unless they open the side nav bar and click home. Having a way for users to easily go back and forth between details for a specific event and overview of all events will be useful.


  19. [H2 Match between system and the real world] (Severity 2) (Evaluators 1)

    After clicking on a specific event to view, I realized there is a bell icon in the top right. I believe there should be context-sensitive help in the form of the user being able to hover over the bell and get information on what it means. I know the bell usually means notifications but I am not sure because your ‘company’ may have a different meaning behind the bell icon.


    Home/Favorited Events

  20. [H4 Consistency and Standards] (Severity 1) (Evaluators 3)

    The Sort/Filter button has a search icon, which may be misleading, as the button also says it is to Filter/Sort. Replacing it with a filter icon might be more intuitive and including the search bar somewhere more readily accessible might help users access specific projects faster, as described in the issue in the General section above.


  21. [H3 User Control and Freedom] (Severity 3) (Evaluators 3)

    There is no way to close or minimize the filters menu; it might help to include an X icon or allow the user to click on the Sort/filter button again to minimize it.


  22. [H7 Flexibility and Efficiency of Use] (Severity 1) (Evaluators 2)

    It might be useful to include an export to iCal button or some other button that allows for external sharing. This way users can add events to their calendars or share events with their friends or teammates.


  23. [H4 Consistency and Standards] (Severity 4) (Evaluators 1)

    In the Favorites screen, clicking the search options sends the user back to the home page. Thus, users are unable to search for events within their favorites catalog, and it’s inconsistent overall as the perceived action and actual action differ (you would think it would open the search options within the favorites screen itself). There is a similar problem in the Subscriptions screen.


  24. [H3 User Control & Freedom] (Severity 2) (Evaluators 1)

    Within the search bar filters, I expect to see a filter group based on what kind of event it is. I believe that should be one of the most important filters.


  25. [H6 Consistency and Standards] (Severity 1) (Evaluators 1)

    The location of certain elements, such as “Events this Month” and the search bar shifts slightly between pages. The title of each page also has different heights. Just stick with one height. In the subscriptions page, the upcoming event items are not properly aligned vertically.


  26. [H4 Consistency and Standards] (Severity 1) (Evaluators 1)

    The filter categories look a bit like buttons, and the “>” symbol makes it seem as though the category titles themselves can be interacted with. Deleting the gray rectangle behind each title might help make it more apparent that the titles are not buttons.


  27. [H4 Consistency and Standards] (Severity 4) (Evaluators 1)

    The filters have no affordances that indicate that the user can interact with them. I would use checkboxes or radio buttons to invite the user to change the selection based on their preferences.


  28. [H5 Error prevention] (Severity 0) (Evaluators 1)

    On the homepage, I think there should be space for one more event on the bottom row as it will be aesthetically pleasing. Not to mention, keeping a constraint of 8 events on the homepage will not contribute to information overload of too many events being shown at once. For instance, Gmail has a constraint of 50 emails showing at a time to prevent information overload.


  29. [H8 Aesthetic and minimalist design] (Severity 0) (Evaluators 1)

    Looking at the “Events this Month” page, I think the number of Event boxes on the upper part of the page should match the number of Event boxes on the lower part of the page, so the design looks consistent (will help with internal consistency).


  30. [H1 Visibility of system status] (Severity 4) (Evaluators 1)

    After clicking the link to enter the Figma prototype, I was immediately greeted with the option to “Login”, but I didn’t find any signifers that I was on the homepage. Since there is no primary navigation bar that shows “about us”, or other action/task oriented buttons on the top, I would think I somehow skipped entering the homepage. On the homepage, there should be a primary navigation bar with information to learn more about Events Newsletter. In regards to Jakob’s law, the homepage of many websites have primary navigation bars and sliders to show the user they are on the homepage.


    Subscriptions

  31. [H7 Flexibility and Efficiency of Use] (Severity 1) (Evaluators 1)

    Including the location, date, and time of an organizer’s upcoming events would allow the user to immediately determine whether or not they could make it to an event. Including a thumbnail of image for the event might also make the section less text-heavy and draw in the user.


  32. [H1 Visibility of System Status] (Severity 3) (Evaluators 1)

    There is no section title for each organizer’s upcoming events, which might be a bit confusing. Including a title like “Upcoming Events” before listing the events might be helpful.


  33. [H6 Consistency] (Severity 1) (Evaluators 1)

    On the subscriptions page, there is a random heart for organization 4 which blocks text, and only for organization 4.


  34. [H6 Recognition rather than recall] (Severity 4) (Evaluators 1)

    Going to the subscriptions page, I am confused as to what type of subscriptions the prototype is referring to, is it referring to paid subscriptions? There should be context perhaps with a question mark signifier that explains the subscriptions page. Alternatively, it might help to change the name of the page to something like “Following,” if that seems to more aptly fit the purpose of the page.


  35. [H4 Consistency and standards] (Severity 3) (Evaluators 1)

    For subscriptions (screen 9), it lists the organizer names that the user is following to see upcoming events. However, when creating an event, it doesn’t ask for the organizer name and only asks for a club name. Users won’t be able to see upcoming events for an organizer they subscribed to if events aren’t connected to organizers.


  36. [H7 Flexibility and efficiency of use] (Severity 2) (Evaluators 1)

I think there should be an option to sign up for a mailing list, in case the user wants to receive notifications of upcoming events or recommendations for events they may like. Similarly, Youtube channels can ask their users to subscribe and turn on notifications to see when their creator uploads.


Recommendations

  1. Fix consistency and standards violations by making the design of the website use more conventionally use icons

  2. Add more shortcuts for users to find events and get notified of them

  3. Implement more undoability after the user clicks buttons

  4. Make it apparent the website has the option for single sign on

James’s Heuristic Evaluation

Home Page

  1. [H3 User Control & Freedom] (Severity 3)

    When clicking the sidebar to the left, the button to close that sidebar (a user’s emergency exit) does not work. On the home page, when the user clicks on sort/filter, there is no option to close the search options that come up. Clicking the sidebar also changes the page that the user is on.

  2. [H4 Consistency] (Severity 1)

    The location of certain elements, such as “Events this Month” and the search bar shifts slightly between pages. The title of each page also has different heights. Just stick with one height. In the subscriptions page, the upcoming event items are not properly aligned vertically.

  3. [H3 User Control & Freedom] (Severity 2)

    Within the search bar filters, I expect to see a filter group based on what kind of event it is. I believe that should be one of the most important filters.

  4. [H6 Recognition rather than recall] (Severity 3)

    While I believe the Sort/Filter icon can be useful in helping users find events, I believe some new users may mistake that button for sorting between what events you are currently registered for. I believe there

    should be a search/explore page that can be accessed within the side menu, especially as I believe this is one of the primary workflows of this site that needs to be better emphasized.

  5. [H7 Flexibility and efficiency of use] (Severity 2)

    Within the search bar, there should be keywords from the user’s search history. This is because users are likely to revisit the same event page to verify times and locations; such a shortcut would greatly increase flexibility and efficiency of use.


    Create Events

  6. [H3 User Control & Freedom] (Severity 3)

    In the Create Events page, there is no cancel button, and if a user wants to get out of this screen, they have to open the sidebar, which again, sends them back to the home page.

  7. [H1 Visibility of system status] (Severity 3)

    After publishing or RSVPing an event, I expect to see some kind of confirmation message. Whether this is a “You have now registered” message with an ability to add the event to your Google Calendar, or a “Check out your published event now!”, there needs to be some kind of feedback mechanism.

  8. [H5 Error prevention] (Severity 3)

    In the Create an Event screen, it would make sense to have required fields indicated by red asterisks, while also having hints to clarify that text is supposed to be entered into the “What is it? And “When/Where.”


    Favorites Screen

  9. [H4 Consistency] (Severity 4)

    In the Favorites screen, clicking the search options sends the user back to the home page. Thus, users are unable to search for events within their favorites catalog, and it’s inconsistent overall as the perceived action and actual action differ (you would think it would open the search options within the favorites screen itself). There is a similar problem in the Subscriptions screen.

  10. [H8 Aesthetic and minimalist design] (Severity 1)

    In the favorites screen, I am not sure how necessary a description of an event under its image will be. This is especially true if the description is long, leading to an eyesore of four paragraphs hanging next to each other. Instead, simply including the title in bigger text below the event image will be simpler and convey the same information if a “view more” tab is added below the title.


    Subscription Page

  11. [H4 Consistency] (Severity 1)

On the subscriptions page, there is a random heart for organization 4 which blocks text, and only for organization 4.


What is done well:

  • Having a favorites screen; increases efficiency for experienced users that go on the site to look at an event they’ve seen on this site before.

  • Strong use of organisms of standalone reusable and portable components. This includes information listed for organizers in the subscription page, the events listed in the favorites tab, and events listed in the “Events this Month” screen.

  • Emphasizing important affordances like “RSVP Here” and “PUBLISH” through color and size.

  • Few buttons overall, where this simple design supports Heuristic rule #9 of minimalistic design.
